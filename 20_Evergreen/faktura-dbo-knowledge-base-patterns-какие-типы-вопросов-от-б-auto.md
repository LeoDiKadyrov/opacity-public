# DBO-внедрения: кластеры повторяющихся вопросов и структура ChromaDB

Опыт 70+ внедрений ДБО говорит одно: 80% вопросов банков повторяются, потому что банки решают одни и те же интеграционные узлы по-разному. Knowledge base имеет смысл только если структурирован по этим узлам, а не по тикетам.

**Шесть кластеров, которые повторяются** (паттерны банковских интеграций; точные пропорции валидируются на реальной выгрузке тикетов):

1. **Аутентификация и RBAC.** SSO через LDAP/AD, mTLS-сертификаты, ролевая модель в УЦ банка. Вопросы — как замаппить роли Faktura на внутренние группы AD, как обновлять сертификаты без даунтайма, что делать с тестовыми контурами. Первый месяц внедрения у большинства.

2. **Регуляторика.** 115-ФЗ (AML/KYC), 161-ФЗ (НПС), валютный контроль, отчётность ЦБ — 440-П, 421-П. Вопросы — как маппить статусы клиентов под антиотмыв, формат выгрузки реестров, обработка стоп-листов Росфинмониторинга. Тянется до прода и периодически вскрывается заново после писем ЦБ.

3. **Платёжные сценарии.** СБП, валютные переводы, корсчёт-операции, цифровой рубль. Вопросы — обработка частичного успеха, ретраи при таймаутах, идемпотентность по ExternalID, маршрутизация через корсчета. Самый дорогой кластер на проде — здесь стоят реальные деньги.

4. **Интеграция с АБС.** ЦФТ-Банк, Diasoft FA, RS-Bank, инхаус. Вопросы — формат проводок, синхронизация остатков, окно регламентных операций, конфликты блокировок. Зависит от вендора АБС больше, чем от Faktura.

5. **Лимиты, комиссии, тарифы.** Иерархия лимитов клиент/договор/продукт, тарифные сетки, расчёт комиссий. Вырастает на пилоте, когда банк впервые видит реальные расчёты.

6. **Reporting и выгрузки.** ОФР, реестры платежей, кастомные отчёты для бизнеса банка. Недооценён в presale, добавляет 20-30% работы в финальной фазе.

**ChromaDB-структура.**

Коллекции — по кластерам, не по банкам. Один банк = метаданные, не отдельная коллекция. Иначе кросс-кейс-поиск сломан.

Метаданные на каждый chunk:
- `cluster` (один из шести)
- `phase` (presale, pilot, prod, регресс)
- `abs_vendor` (для кластера 4)
- `regulator_ref` (номер ФЗ/положения)
- `bank_size` (sm/md/lg — паттерны вопросов зависят от размера)
- `resolution_type` (config, code, doc, escalation)

Чанкинг — семантический ~512 токенов, overlap 50. Тикеты не атомарны: вопрос, дискуссия, решение — три разных эпистемических куска. Чанковать раздельно, иначе ретриверу подаются нерелевантные фрагменты.

Эмбеддинг — multilingual: BGE-M3 или e5-large-multilingual. OpenAI ada-002 на русском техническом языке проседает на 10-15% recall.

Hybrid search обязателен: dense + BM25 + metadata filter. Чистый vector search ломается на технических ID (номера положений, коды ошибок, артикулы продуктов) — там нужен exact match.

**Где это сломается.**

- Копирование тикетов "как есть" → свалка. Нужен upstream-процесс: автор тикета помечает cluster+resolution_type перед закрытием. Без этого retrieval mediocre навсегда.
- Кластер 2 (регуляторика) устаревает быстрее остальных. TTL + periodic re-validation против актуальных писем ЦБ.
- Цифровой рубль — отдельный кластер 7, не подкластер платежей. Логика расчётов и роль платформы ЦБ структурно другие.

**Key claims:**

1. ChromaDB hybrid search через `where` filters + dense embeddings — официальная документация. [docs.trychroma.com](https://docs.trychroma.com/)
2. BGE-M3 превосходит OpenAI ada-002 на multilingual MTEB. [arxiv.org/abs/2402.03216](https://arxiv.org/abs/2402.03216)
3. 115-ФЗ — основа AML-вопросов в DBO. [consultant.ru — 115-ФЗ](https://www.consultant.ru/document/cons_doc_LAW_32834/)
4. 161-ФЗ — основа НПС-вопросов. [consultant.ru — 161-ФЗ](https://www.consultant.ru/document/cons_doc_LAW_115625/)
5. Цифровой рубль — платформа ЦБ. [cbr.ru/fintech/dr](https://www.cbr.ru/fintech/dr/)
