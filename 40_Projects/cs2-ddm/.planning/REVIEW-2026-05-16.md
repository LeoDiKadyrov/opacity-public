## BLOCKER findings

### B-1: `T1_GRACE_MS = 120` creates 125ms hard floor on T0→T1 (CONFIRMED — trigger bug)

**Files:** `config.py:124`, `ddm_analyzer.py:517-518`
**Status:** In-experiment as of brief creation (grace=30 vs grace=0 underway).

**Mechanism (verified by code trace):**

```python
# ddm_analyzer.py:517-518
grace_ticks = int(T1_GRACE_MS / (1000 / self.tickrate))  # int(120/15.625) = 7
aim_search_start = t0_tick + grace_ticks
```

Inside the loop, `potential_t1 = int(nxt["tick"])` where `nxt = iloc[i+1]`. At `i=0`: `curr=t0+7`, `nxt=t0+8`. Minimum T1 delta = 8 ticks = **125ms**. Matches DB distribution evidence exactly.

**Redundancy proof:** `_detect_t1` already runs three filters that prevent the failure mode grace was meant to prevent ("pre-aim micro-corrections as T1"):

|Filter|Line|What it prevents|
|-|-|-|
|`sig_change` (`d_yaw > 0.08° OR d_pitch > 0.08°`)|552|jitter / sub-degree drift|
|`moving_towards` (`nxt_dist < curr_dist - 0.01°`)|562|non-reactive corrections away from enemy|
|`sustained` (`T1_SUSTAINED_AIM_TICKS=2` consecutive)|578|single-tick noise spikes|

Grace stacked on top = hard floor masquerading as physiology. 1145 engagements pinned at exactly 8-tick delta in the DB.

**Recommendation:** REMOVE the grace gate entirely (`T1_GRACE_MS = 0`). Keep semantic chain intact. If a regression appears in T1 noise, raise `T1_SUSTAINED_AIM_TICKS` to 3 or tighten `T1_MIN_ANGLE_CHANGE` — both semantic, neither creates a structural cliff.

**Test changes required:**
- `tests/test_ddm_analyzer_t1.py::test_t1_grace_period_excludes_early_ticks` (lines 291-305) asserts the *buggy* behaviour. Replace with a test confirming early aim ticks pass through and feed into sustained-aim detection.
- Add `tests/test_distribution_shape.py` regression suite (spec in section below).

**Downstream re-derivation needed:**
- `interpretation.py:_FALLBACK_THRESHOLDS["peek"]["rt_visible_to_aim_ms"][0] = 125.0` — currently equals the floor; stale post-fix.
- `interpretation.py:_ABSOLUTE_ELITE_CEILING["peek"]["rt_visible_to_aim_ms"] = 200.0` — derived from floor-clipped data; re-derive p25 from clean corpus.
- All landing claims under "Reaction Time" → re-batch + recalculate.

### B-2: `DuelAttemptFinder` does NOT gate on player `is_alive` (Bug C class — phantom-after-death)

**Files:** `t0_detector.py:209-289` (`find_visible_enemies_at_tick`), `t0_detector.py:291-316` (`find_first_visible_enemy_in_window`), `duel_attempts.py:180-185` (consumer).
**Severity:** BLOCKER for kill_rate / hit_rate aggregates.

The `engagements` table got the Bug C fix on 2026-05-14: `analyze_engagement_episode` checks `player_death_ticks` against `t0_tick` within the same round (`ddm_analyzer.py:679-695`). The `duel_attempts` table never got the same fix.

`DuelAttemptFinder.find_attempts` calls `t0_detector.find_first_visible_enemy_in_window` (line 180) with no `player_death_ticks` argument. `find_visible_enemies_at_tick` gates only on **enemy** `is_alive` (line 253-255), not on player. If the player's tick row persists after their death (spectator/corpse frames), the BVH ray geometry test still runs and can fire T0 on a teammate's enemy.

**Evidence pattern:** Pre-fix 2026-05-14, donk's row 400 in `mouz-vs-spirit-m3-overpass.dem` round 15 fired T0 at tick 131757 after donk died at 129869. Engagements path now rejects this. Attempts path still accepts it. Kill rate denominator inflated by ~3-8% per player (proportional to deaths-per-round × spectator-tick persistence).

**Recommendation:**

```python
# duel_attempts.py — DuelAttemptFinder.__init__ accepts player_death_ticks
# DuelAttemptFinder.find_attempts threads them into _process_cluster
# In _process_cluster, after find_first_visible_enemy_in_window returns t0_tick:

if self._player_dead_at(t0_tick, round_start_ticks, player_death_ticks):
    return None  # phantom-after-death
```

Or pass `player_death_ticks` into `find_first_visible_enemy_in_window` and filter inside the loop. Same semantic; pick based on which layer should own the gate.

Backfill: requires re-batch of duel_attempts (or targeted DELETE for rows where `t0_tick > player_death_tick` in same round, but recomputing in-place is safer).

### B-3: `DuelAttemptFinder` does NOT gate on flash blindness

**Files:** `t0_detector.py:209-316`, `duel_attempts.py:180-185`.
**Severity:** BLOCKER for duel_attempts data quality.

`find_t0` (engagements path) skips flash-blinded ticks via `flash_intervals` (line 154-156). `find_first_visible_enemy_in_window` (attempts path) does not accept the kwarg and does not skip flashed ticks.

Result: a player flashed when they start firing into a contested angle gets a T0 fixed at the moment the geometry clears — even though they couldn't see the screen. Attempts table records a "reaction" with bogus crosshair_angle and bogus first-burst stats.

**Recommendation:** Add `flash_intervals` kwarg to `find_first_visible_enemy_in_window` + propagate from `DuelAttemptFinder._process_cluster`. Parse via `T0Detector.parse_flash_intervals` once per demo and thread through (already done in `ddm_analyzer.py:388-393`).

### B-4: T1 silently dropped when player already on target at T0 (selection bias)

**File:** `ddm_analyzer.py:564`
**Severity:** BLOCKER — survivorship bias inverted (the best pre-aims drop out).

```python
if sig_change and moving_towards and curr_dist > T1_NOT_AIMED_THRESHOLD:
```

If player crosshair is already within `T1_NOT_AIMED_THRESHOLD = 1.0°` of the enemy at every tick from T0+grace to T2, the predicate is always false, `consecutive` never advances, T1 returns -1, `rt_visible_to_aim_ms = NaN`. The engagement is *kept* in the table (good) but the metric is censored.

This is exactly the class of engagement where T0→T1 should be **zero or near-zero** — the player was pre-aimed; perception+motion-planning was complete before visibility. By censoring it, the aggregate T0→T1 distribution loses its left tail. Median rises. The metric stops measuring what it claims.

**Recommendation:** Branch on `curr_dist ≤ T1_NOT_AIMED_THRESHOLD` at T0+grace:
- If already on target AND no significant correction in the first N ticks → set `T1 = T0`, `rt_visible_to_aim_ms = 0`.
- Else continue current logic.

Add an explicit per-engagement flag `t1_source ∈ {"sustained_aim", "pre_aimed"}` so the chart can split or pool as needed. Coaching narrative can read it.

**Note on B-1 + B-4 interaction:** Both must be fixed together. Fixing only B-1 (remove grace) still leaves B-4 censoring the pre-aim cases. Fixing only B-4 still leaves B-1 floor at 125ms for non-pre-aimed engagements. Order does not matter, but they ship as one PR.

## WARN findings

### W-1: `_HIT_WINDOW_TICKS = 192` is dead code

**File:** `config.py:66`. Grep confirms zero non-config usage. Delete the constant + comment.

### W-2: `_HIT_LATENCY_TICKS = 4` lives outside `config.py` + is a tick-quantum value

**File:** `duel_attempts.py:23`. Constant = 62.5ms exactly (4 ticks × 15.625ms). Move to `config.py` next to the other duel-attempts knobs and document the choice. Tick-quantum values are smell-test candidates for the same floor pattern as B-1; here it controls the bullet→hit-association window so a 60-65ms window is reasonable, but the comment-only justification is weak. Bonus: enables global tickrate change without missing the constant.

### W-3: `T1_MOVING_TOWARDS_TOLERANCE = 0.01°` is below demo angular quantization noise

**File:** `config.py:137`, used at `ddm_analyzer.py:562`.

demoparser2 yaw/pitch quantization is 16-bit (~0.0055°/step rounded to 0.011° in practice). Tolerance 0.01° is at the noise floor. Effective filter strength ≈ 0. The "moving_towards" predicate becomes "any non-zero inward step", which is what `sig_change` (≥0.08°) already does.

**Recommendation:** Either remove `moving_towards` (already covered by `sig_change`+`sustained`), or raise tolerance to ~0.05–0.10° to make it semantically distinct from `sig_change`. Linked anti-pattern: redundant filter stacking.

### W-4: `T1_NOT_AIMED_THRESHOLD = 1.0°` triple-counts and creates B-4

**Files:** `config.py:134`, `ddm_analyzer.py:564`.

Already covered under B-4. Marked WARN here for redundancy-audit completeness: the third predicate piles onto `sig_change + moving_towards`, replicating the B-1 anti-pattern at angular scale.

### W-5: `pd.read_sql` used on tables containing `player_steamid` columns

**File:** `interpretation.py:211, 224, 240, 298, 305`.

Each call queries `engagements` or `duel_attempts` with `WHERE player_steamid = ?`. None SELECT the player_steamid column → currently safe. CLAUDE.md rule is stricter: "NEVER use `pd.read_sql` on tables with `player_steamid`". Future SELECT addition (e.g., debug column) silently truncates. Migrate to `cursor.fetchall()` + manual dict for consistency with `report_generator.py` (which got it right).

### W-6: `multi_player_analyze.extract_steamids` parses entire demo for roster lookup

**File:** `multi_player_analyze.py:62`. Docstring claims "~first 6400 ticks". Code: `parser.parse_ticks(["steamid"])` — no `ticks=` kwarg → full demo parse. Fix:

```python
df = parser.parse_ticks(["steamid"], ticks=list(range(6400)))
```

10-30× speedup on the wrapper's startup cost. No correctness change.

### W-7: `astype("int64")` after `dropna` on SteamID column is fragile

**File:** `multi_player_analyze.py:67-68`.

If `steamid` column contains a single NaN, pandas promotes the column to float64 *before* `dropna()` runs. The 17-digit precision is then irrecoverable even after `dropna().astype("int64")`. Replace with:

```python
sids = (
    pd.to_numeric(df["steamid"], errors="coerce")
    .dropna()
    .astype("int64")
    .unique()
    .tolist()
)
```

Same pattern used safely in `ddm_analyzer.py:906-941`.

### W-8: `_compute_velocity` single-tick delta is noisier than CS2 movement (Bug B root)

**Files:** `ddm_analyzer.py:455-473`, `duel_attempts.py:279-296`.

Velocity computed from one-tick (15.625ms) XY delta × tickrate. Sub-tick demo quantization gives wildly varying instantaneous speeds. This is the mechanism behind Bug B (strafe-hold mis-classified as peek): a player holding an angle while strafing micro-side-to-side gets snapshot at the high-velocity half of a cycle.

**Recommendation:** Average position delta over a 4-tick window (60ms) centered on T0:

```python
def _compute_velocity(self, steamid, t0_tick, all_ticks_df, label=""):
    rows = all_ticks_df[
        (all_ticks_df["steamid"] == steamid)
        & (all_ticks_df["tick"].between(t0_tick - 2, t0_tick + 2))
    ].sort_values("tick")
    if len(rows) < 2:
        return np.nan
    dx = float(rows.iloc[-1]["X"]) - float(rows.iloc[0]["X"])
    dy = float(rows.iloc[-1]["Y"]) - float(rows.iloc[0]["Y"])
    n_ticks = int(rows.iloc[-1]["tick"]) - int(rows.iloc[0]["tick"])
    return math.sqrt(dx*dx + dy*dy) * self.tickrate / max(n_ticks, 1)
```

Combined with W-12 (position-stability cross-check) this resolves Bug B.

### W-9: Smoke modelled as sphere; CS2 smokes are oblate ellipsoid

**File:** `t0_detector.py:355-359`.

`dist_sq <= _SMOKE_RADIUS_UNITS ** 2` is a sphere check. CS2 smokes are taller than wide (~144u radius, ~200u tall). Rays from above (jump-peek, jumpshot at A-long Ancient, B-tunnels Inferno) may clear the modelled sphere while the in-game smoke still blocks vision. Likely small effect on bulk metrics; document as a known limit on the methodology page.

### W-10: 300-tick overlap-window magic number

**File:** `ddm_analyzer.py:1043`. Used to reject same-target re-engagements within ~4.7s of last accepted T2. Extract to `config.py` as `_OVERLAP_WINDOW_TICKS = 300` with comment.

### W-11: `is_spotted` requested in `_TICKS_OPTIONAL` despite being broken in GOTV

**File:** `ddm_analyzer.py:52`. CLAUDE.md, PITFALLS.md, and ARCHITECTURE.md all document that `m_bSpotted` / `is_spotted` is never populated in CS2 GOTV demos. Listed as an optional parse_ticks column. demoparser2 will silently fill nulls — wasted bandwidth and zombie reference. Anti-pattern #4 violation. Remove.

### W-12: `engagement_type` derived from velocity alone (Bug B, restated for cross-reference)

**Files:** `ddm_analyzer.py:585-589`, `duel_attempts.py:192-196`. Already documented inline as TODO at `ddm_analyzer.py:743`. Fix sketch:

```python
def _classify_engagement(self, velocity_ups, position_stability_ups):
    # peek requires BOTH velocity above threshold AND position changed enough
    if velocity_ups >= self.player_velocity_threshold and position_stability_ups > 32.0:
        return "peek"
    return "hold"

# position_stability_ups computed as max(|X|, |Y|) variance over [T0-30, T0]
# Strafing-hold has high instant velocity but ~zero positional drift → hold.
```

## Distribution-shape regression spec (new test class)

The T1_GRACE bug was undetectable in unit tests because unit tests check single-engagement outputs. Aggregate floor artifacts only surface against the post-batch DB. Add a fourth test class (alongside core / geometry / quality / t1) that runs against a populated `analytics.db`.

**Proposed file:** `tests/test_distribution_shape.py`
**Marker:** `@pytest.mark.requires_db` — skip if `analytics.db` missing or N<1000.

### Checks

For each `(player_steamid, engagement_type, metric)` group where N ≥ 100:

```python
def test_no_value_dominates(df, metric):
    """No single tick-quantum value > 5% of samples."""
    counts = df[metric].value_counts()
    mode_pct = counts.iloc[0] / len(df)
    assert mode_pct < 0.05, (
        f"{metric} mode value {counts.index[0]:.1f} pinning {mode_pct:.1%} of N={len(df)} — "
        "suspect floor/ceiling artifact"
    )

def test_min_not_floor_pinned(df, metric, tick_ms=15.625):
    """Min should not equal an exact tick multiple if that value also dominates p10."""
    m = df[metric].min()
    p10 = df[metric].quantile(0.10)
    if abs(m % tick_ms) < 0.5 and abs(m - p10) < 0.1:
        pytest.fail(
            f"{metric}: min={m:.3f} and p10={p10:.3f} both equal a tick multiple → "
            "floor artifact (B-1 class)"
        )

def test_median_strictly_above_min(df, metric, slack=1.5):
    """Median should be > slack × min for any continuous reaction metric."""
    m = df[metric].min()
    med = df[metric].median()
    assert med > slack * m + 1, (
        f"{metric}: median {med:.1f} ≤ {slack}× min {m:.1f} → distribution collapsed"
    )
```
