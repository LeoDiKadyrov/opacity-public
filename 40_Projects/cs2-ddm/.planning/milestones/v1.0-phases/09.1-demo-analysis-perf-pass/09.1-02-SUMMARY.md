# Phase 9.1 Plan 02: Batch parse_events to Single Rust Pass Summary

One-liner: Replaced 4 separate `parse_event()` calls in `analyze_demo()` with a single batched `parse_events([...])` call, converted via `dict(by_name)` for non-deterministic order safety (Pitfall #2). Empirically measured ~4.97× speedup on multi-event extraction.

## Changes

`ddm_analyzer.py:743-797`:
- Removed 4 separate `self.parser.parse_event("<name>")` calls (weapon_fire, player_hurt, player_death, round_start).
- Added single `self.parser.parse_events(events_to_parse)` call with `events_to_parse: List[str] = ["player_hurt", "player_death", "weapon_fire", "round_start"]`.
- Coerced result via `by_name: Dict[str, pd.DataFrame] = {name: df for name, df in raw_events}` immediately — never positional-unpack (Pitfall #2).
- Each downstream variable (`all_hurt_df`, `all_death_df`, `fire_df_all`, `rs_df`) assigned via `by_name.get("<name>", pd.DataFrame(columns=["tick"]))`.
- Per-player `weapon_fire` filtering stays in Python post-parse (Pitfall #6).
- Whole batch wrapped in try/except → empty fallback dict on parser exception (preserves graceful degradation that the old per-event try blocks provided).
- Removed redundant inner try/except around `round_start` extraction (frame already in `by_name`).

**2. [Rule 2 — Critical functionality] Try/except around the whole batch**
- **Issue:** Old code had 4 individual `try/except` blocks (one per parse_event) → graceful fallback to empty frames if any single event parse failed. Naive batch refactor would lose that.
- **Fix:** Wrapped `parse_events(...)` + `dict(by_name)` in a single try/except. On exception: log warning, set `by_name = {}` so subsequent `.get(name, default)` calls produce empty frames — same end-state as before. Preserves correctness contract.
