#### `parse_ticks(wanted_props, *, players=None, ticks=None, prop_states=None)`

```
Signature (Rust):
    pub fn parse_ticks(
        &self,
        wanted_props: Vec<String>,
        players: Option<Vec<u64>>,
        ticks: Option<Vec<i32>>,
        prop_states: Option<Vec<WantedPropState>>,
    ) -> PyResult<Py<PyAny>>
```

| Parameter | Accepts | Behavior |
|-|-|-|
| `wanted_props` | `list[str]` | required |
| `ticks` | `list[int]` OR `range` | **filters during parsing** (`wanted_ticks` in `ParserInputs`). NOT a set. |
| `players` | `list[int]` (u64) | **BUG observed**: `players=[<SteamID64>]` returned 0 rows on test demo. Treat as broken — do post-parse `df[df.steamid==sid]` instead. [VERIFIED empirically] |
| `prop_states` | `list[WantedPropState]` | not used in this phase |

**Critical gotchas:**
- `ticks=[]` (empty list) returns **FULL parse** (not zero rows) because `unwrap_or_default()` on Rust side gives empty Vec which the parser treats as "no filter". **MUST guard:** `if not ticks: skip selective branch`.
- `set` raises: `argument 'ticks': 'set' object cannot be cast as 'Sequence'`. Always pass list/range.
- `range()` works (PyO3 sequence protocol).
- Returned object: dict-like compatible with `pd.DataFrame(...)` constructor (current code does this).

**Measured perf (1.97M-row demo):**

| Call | Time | Rows |
|-|-|-|
| `parse_ticks(['X','Y','Z'])` (full) | 2.99s | 1,976,630 |
| `parse_ticks(['X','Y','Z'], ticks=range(10000,10500))` | 0.61s | 5,000 |
| `parse_ticks(['X','Y','Z'], ticks=<20×500-tick windows>)` | 0.75s | 99,980 |

#### `parse_events(event_name, *, player=None, other=None)`

```
Signature (Rust):
    pub fn parse_events(
        &self,
        event_name: Vec<String>,
        player: Option<Vec<String>>,
        other: Option<Vec<String>>,
    ) -> PyResult<Py<PyAny>>
```

**Returns:** `List[Tuple[str, pandas.DataFrame]]` — list of `(event_name, df)` pairs. Order **not guaranteed to match input order** (observed: requested `['player_hurt','player_death','weapon_fire','round_start']`, got `[('round_start',...), ('weapon_fire',...), ('player_hurt',...), ('player_death',...)]`). **Always convert to dict by name** before use:

```python
results = parser.parse_events(["player_hurt", "player_death", "weapon_fire", "round_start"])
by_name = {name: df for name, df in results}
hurt_df  = by_name.get("player_hurt", pd.DataFrame())
death_df = by_name.get("player_death", pd.DataFrame())
```

**Schema difference vs `parse_event` (singular):**

| Method | Returns |
|-|-|
| `parse_event("player_hurt")` | DataFrame directly |
| `parse_events(["player_hurt"])` | `[("player_hurt", DataFrame)]` |

The DataFrames themselves have **identical columns** (verified: hurt → `armor, attacker_name, attacker_steamid, dmg_armor, dmg_health, health, hitgroup, tick`).

**Measured perf:** 4× `parse_event(...)` separately = 3.38s; one `parse_events([4])` = 0.68s (**4.97×**).

## Architecture Patterns

### System Data Flow (Phase 9.1 hot path)

```
analyze_demo(profile=True)
  │
  ├─[T0_step1: build hurt anchors]── parse_events(["player_hurt"]) ─┐
  │                                                                  │
  │  (need anchors BEFORE selective parse_ticks — Rule of Order)     │
  │                                                                  │
  ├─[T0_step2: full event batch]──── parse_events([4 events])  ─────┤
  │   (player_hurt, player_death, weapon_fire, round_start)          │
  │                                                                  │ ← #2
  ├─[T0_step3: build tick window union]                              │
  │   union = np.unique(np.concat([arange(h-W, h+W) for h in hurts]))│
  │   if PARSE_TICKS_SELECTIVE and len(union)>0:                     │
  │     all_ticks_df = parse_ticks(props, ticks=union)  ←─── #3      │
  │   else:                                                           │
  │     all_ticks_df = parse_ticks(props)  (current path)            │
  │                                                                  │
  ├─[T0_step4: build per-sid cache]  ←─────────────────────────────  │
  │   ticks_by_sid = {sid: g.sort_values("tick")                     │
  │                    for sid,g in df.groupby("steamid",sort=False)} #4
  │                                                                  │
  ├─[T0_step5: per-engagement loop]                                  │
  │   for moment in self.analysis_moments:                           │
  │       analyze_engagement_episode(..., ticks_by_sid=ticks_by_sid) │
  │           └── find_t0(..., ticks_by_sid=ticks_by_sid)            │
  │               └── if ticks_by_sid: p_df = ticks_by_sid[pid]      │
  │                   else: p_df = df[df.steamid==pid] (fallback)    │
  │                   for tick in p_df.iter:  ──── center ray FIRST  │
  │                       for offset in _AABB_OFFSETS:               │ #1
```

**Critical ordering note:** parse_events MUST run **before** selective parse_ticks because the union of selective windows is built from `player_hurt.tick` values. Means commits land in this dependency order: #2 enables data for #3.

### Anti-Patterns to Avoid

- **`parser.parse_ticks(props, ticks=[])`**: silently returns full parse. Always guard with `if len(window) > 0`.
- **`parser.parse_ticks(props, players=[<sid>])`**: returns 0 rows for u64 SteamID64 — broken. Do post-parse filter.
- **Assuming `parse_events` preserves input order**: it doesn't. Build `dict(by_name)` first.
- **Mutating `ticks_by_sid[sid]` in callees**: cached frames must stay read-only — callees should `.copy()` if they intend to mutate. Use `df.loc[mask]` (returns view) carefully.
- **Calling `groupby` inside the engagement loop**: defeats the cache. Build once в `analyze_demo()`.
- **Empty-list `ticks=` as "no anchors"**: silently slow. Branch on `len(anchors) == 0` before the parse_ticks call.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|-|-|-|-|
| Multi-event batching | manual threading / multiprocess of `parse_event` | `parse_events([list])` | Rust-side single-pass, ~5× faster, no thread overhead |
| Tick window filtering | post-parse `df[df.tick.isin(set)]` | `parse_ticks(ticks=union)` | filters in Rust before DataFrame materialization, ~5× faster on narrow windows |
| Per-steamid filtering for u64 SteamID64 | `parse_ticks(players=[sid])` | post-parse `df[df.steamid==sid]` or **groupby cache** | `players=` returns 0 rows for u64 SIDs; broken |
| Visibility batched check | numpy-vectorized AABB raycast | (deferred — single `is_visible` for now) | awpy 2.0.2 has no batch API; would need C/PyO3 work |
| Per-step timer harness | wrapping every line with logger | `time.perf_counter()` deltas saved to dict, JSON-dumped | minimal overhead, structured output |

## Common Pitfalls

### Pitfall 1: `ticks=[]` silently returns full parse
**What goes wrong:** if `player_hurt` events list is empty (rare but possible — bot demos, very short demos), the union of windows is empty, `parse_ticks(props, ticks=[])` runs the full parse. Test logs "selective parse 5000 rows" for normal demos, then 1.97M rows for an edge case demo, masking a regression.
**Why it happens:** Rust `unwrap_or_default()` on `Option<Vec<i32>>` produces empty `Vec`, which the parser treats as "no filter".
**How to avoid:** explicit guard `if PARSE_TICKS_SELECTIVE and len(window) > 0` before calling with `ticks=`. Add unit test: empty-anchors demo path falls through to full parse, not silent slow path.
**Warning signs:** bench shows correct speedup on most demos but one demo's `parse_ticks` step matches full-parse baseline.

### Pitfall 2: `parse_events` return order is non-deterministic
**What goes wrong:** code unpacks `[(name1, df1), (name2, df2), ...] = results` positionally → wrong DataFrame assigned to wrong variable.
**Why it happens:** Rust HashMap iteration order; observed reordering on probe.
**How to avoid:** **always** convert to `dict(results)` immediately. Never positional unpack.

### Pitfall 3: `parse_events` schema differs from `parse_event` (singular)
**What goes wrong:** mocks in existing 322 tests stub `parser.parse_event.return_value = <DataFrame>`. Switching production code to `parse_events([...])` → mocks no longer intercept the new code path → test fixtures break.
**Why it happens:** different method names, different return shapes (DataFrame vs list-of-tuples).
**How to avoid:** in tests, mock `parser.parse_events.return_value = [("player_hurt", df_hurt), ("player_death", df_death), ...]`. Do this in a Wave-0 test-fixture refactor BEFORE shipping #2 production change. Or: keep one shim helper `_get_events(events)` that calls `parse_events` in production but is patched in tests.

### Pitfall 4: groupby cache + duplicate steamid index
**What goes wrong:** existing `find_t0` does `e_df.set_index("tick")` then `e_df.loc[tick]`, with explicit handling for duplicate ticks (`if isinstance(e_row, pd.DataFrame): e_row = e_row.iloc[0]`). When using cached groupby, ensure the cached frame still has `tick` column (groupby preserves columns by default — verified pandas 3.0.1).
**Why it happens:** if you accidentally do `groupby("steamid").agg(...)` instead of just splitting groups, you lose granularity.
**How to avoid:** use the iterator form `{sid: g for sid, g in df.groupby("steamid", sort=False)}` — `g` is the full sub-DataFrame including all original columns.

### Pitfall 5: tick-window union memory blow-up
**What goes wrong:** ~30 anchors × 600-tick windows = 18,000 ints, but if you use `set.union()` with thousands of `range` objects, Python overhead dominates.
**Why it happens:** repeated set-union in Python is O(n²) for naïve `for r in ranges: result |= set(r)`.
**How to avoid:** `np.unique(np.concatenate([np.arange(h-W, h+W) for h in hurt_ticks])).tolist()` — vectorized, ~10x faster, returns sorted-unique list. Then pass directly to `parse_ticks(ticks=...)`. Measured: union of 30×600=18000 ticks → 0.001s with numpy approach.
**Warning signs:** `_build_tick_window_union` itself shows up as >50ms in profile.

### Pitfall 6: `players=` kwarg looks attractive but is broken for u64 SIDs
**What goes wrong:** "let's also pass players=[player_steamid] to parse_ticks for double speedup" → returns 0 rows → silent crash later when DataFrame is empty.
**Why it happens:** likely a u64 ↔ i32 conversion glitch in PyO3 binding for the `players` param. Empirically returned 0 rows for valid SteamID64.
**How to avoid:** don't use `players=` in this phase. Stick to `ticks=` + post-parse filter. Add a docstring note in the helper.

### Example 2: Tick-window union helper

```python
# ddm_analyzer.py
import numpy as np
from config import T0_MIN_OFFSET_TICKS

# Lookback before player_hurt. ~5s search_start window already used (config:_ATTEMPT_WINDOW_*).
# Keep it generous to avoid clipping legitimate T0; selective gain remains huge.
_SELECTIVE_WINDOW_BEFORE_TICKS: int = 384   # ~6s before each hurt
_SELECTIVE_WINDOW_AFTER_TICKS:  int = 192   # ~3s after

def _build_tick_window_union(self, anchor_ticks: list[int]) -> list[int]:
    """Vectorized union of tick windows around player_hurt anchors.

    Returns sorted unique list[int] suitable for parse_ticks(ticks=...).
    Empty input → empty list (caller guards on len()).
    """
    if not anchor_ticks:
        return []
    arrs = [
        np.arange(t - _SELECTIVE_WINDOW_BEFORE_TICKS,
                  t + _SELECTIVE_WINDOW_AFTER_TICKS,
                  dtype=np.int32)
        for t in anchor_ticks
    ]
    union = np.unique(np.concatenate(arrs))
    union = union[union >= 0]   # demoparser2 ticks are non-negative
    return union.tolist()
```
