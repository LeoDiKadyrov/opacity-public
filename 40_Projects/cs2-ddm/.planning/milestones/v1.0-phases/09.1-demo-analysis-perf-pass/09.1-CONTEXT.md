<prior_decisions>

**From v0.x:**
- BVH+AABB raycast is the only correct T0 approach (m_bSpotted not in GOTV demos; FOV-only fails for peeks). Optimizations MUST preserve "any AABB corner ray = visible" semantics.
- T0 = first tick ANY corner ray returns True. Center ray reordering (change #1) does NOT change semantics — still returns first hit, just checks center first.

**From Phase 6:**
- `T0_MIN_OFFSET_TICKS=20` quality gate (config.py:89) — selective parse_ticks window must respect this offset.

**From Phase 7:**
- ProcessPool batch_runner already parallelizes ACROSS demos. Phase 9.1 optimizes WITHIN one demo. Both layers compose.
- BVH .tri files load once per process (~one-time cost amortized).

**From memory:**
- demoparser2 0.41.2 supports `parse_events([list])` for single-pass multi-event extraction.
- demoparser2 0.41.2 supports `parse_ticks(props, ticks=[list_of_ticks])` for selective windows.

<decisions>

### Benchmark methodology (D-01)

- **Demo set:** 10 demos (subset of 26 donk demos covering ≥3 distinct maps for .tri load variance)
- **Metric:** BOTH walltime end-to-end (SC5 verdict) AND per-step breakdown (`parse_ticks`, `parse_events`, `find_t0` loop) for diagnostic per-change attribution
- **Cache state:** Cold (fresh process per run, .tri loaded each time) — matches batch_runner reality
- **Implementation:** Extend existing `analyze_demo(profile=True)` with per-step `time.perf_counter()` markers. Save baseline + per-change runs to `bench_results.json` for comparison.
- **Why:** Single demo not representative (smoke/flash variance per round). Per-step lets us attribute speedup to individual changes (#1–#4) — protects against false-positive aggregate gains.

### Selective parse_ticks fallback (D-02)

- **Strategy:** Feature flag `config.PARSE_TICKS_SELECTIVE` (default `True`). If regression detected, flip to `False` without revert.
- **Regression definition:** walltime worse than baseline (>0% regression on 10-demo bench)
- **Why:** Multiple parse_ticks calls may have per-call overhead. Feature flag = production-safe toggle without git revert. Keeps full code path intact for re-enable after demoparser2 upgrades.
- **Implementation:** Wrap selective logic with `if PARSE_TICKS_SELECTIVE: <new path> else: <existing full-parse path>`. Test both paths.

### Commit granularity (D-03)

- **Strategy:** 4 atomic commits (one per change), TDD test-first (RED+GREEN per change → 8 commits total + verifier + SUMMARY commits)
- **Order:** #1 (center ray) → #2 (parse_events) → #3 (selective parse_ticks) → #4 (per-steamid cache). Cheap wins first.
- **Why:** Atomic per-task is GSD discipline (322 tests already TDD-built). Easy git revert per optimization. Early wins (#1, #2) deliver value even if #3 or #4 regress.
- **Plans:** ~4 plans (one per optimization) + final benchmark/verification plan.

### Per-steamid DataFrame cache (D-04)

- **Location:** `analyze_demo()` local dict — `ticks_by_sid = {sid: group for sid, group in all_ticks_df.groupby("steamid")}`. Built once, passed down.
- **API change:** `find_t0()` and `analyze_engagement_episode()` accept new optional kwarg `ticks_by_sid: Optional[dict] = None`. If `None`, falls back to current bool-filter on `all_ticks_df` (zero-regression for existing tests/mocks).
- **Why:** Local dict scope = no global state, no ProcessPool leakage risk. Optional kwarg + fallback = mocks in `test_t0_detector_*.py` and `test_ddm_analyzer_*.py` keep passing without rewrite.
- **Risk note:** `groupby` materialization cost vs repeated bool-filter — should be cheaper amortized over ~30 engagements per demo, but bench will confirm.

</decisions>
