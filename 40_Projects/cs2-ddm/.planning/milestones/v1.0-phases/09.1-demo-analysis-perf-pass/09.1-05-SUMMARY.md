## SC5 Diagnosis

**Headline:** The 4 perf changes are correct and not regressed. The 1.08× aggregate is real but small because the optimized code paths are a small fraction of cold-cache `analyze_demo()` walltime.

**Likely walltime distribution per demo (~280-360s):**

- **`.tri` BVH mesh load (DDMAnalyzer construct):** ~26-32s (~10%)
- **`find_visible_enemies_at_tick` BVH raycast loops** (per-tick, NOT in scope of Phase 9.1): ~50-70% of remaining time, dominant
- **demoparser2 `parse_ticks` Rust pass on full demo:** still substantial even with selective windowing because the windows still cover most of an active round
- **`parse_events` batch:** small but real win (~0.5-2s per demo absolute)
- **AABB center-ray first:** changes loop iteration count not loop body; small absolute win
- **`ticks_by_sid` cache:** real but small win — fewer DataFrame bool-filters per `find_t0` call

**Why RESEARCH.md 4.97× isolated didn't translate:** that micro-benchmark measured `parse_events` vs N× `parse_event` in isolation. End-to-end walltime is dominated by other code paths that Phase 9.1 explicitly DEFERRED (per CONTEXT.md "Deferred: find_visible_enemies_at_tick cache — lower priority").

**Recommended follow-up (NOT done in this plan):**

- **New phase 9.2 (or 999.x backlog):** profile `analyze_demo()` end-to-end with `cProfile`/`py-spy` against a spirit demo to identify the actual hot path. Likely candidates: per-tick BVH raycast in `find_visible_enemies_at_tick`, per-tick state coercion in `parse_ticks` post-processing, eye-height geometry computation.
- **Defer** the SC5 ≥3× target until after that profiling — the next gate should be calibrated on measured per-step contribution, not isolated micro-benchmarks.

## Deviations from Plan

### [Rule 1 — D-01 implementation gap, pre-existing]

Plan 00 was supposed to extend `analyze_demo(profile=True)` with per-step `time.perf_counter()` markers per D-01. The signature exists (`profile: bool = False`) but the body never sets `_last_profile`. Bench harness reads `getattr(analyzer, "_last_profile", None) or {}` and gets `{}` → falls back to construct-only timing. This is a Wave 0 gap that wasn't caught by the SC5 RED tests because they only verified the harness produces JSON, not what's IN it.

### [Rule 1 — Wrong corpus first attempt]

First baseline + post-change run (commits `1f5cb88`, `dd702c4`, `6148666`) used the parent `for_analysis/` dir which contains abdra/rachelr/shoke/faze/spirit-mixed/strogo demos but NOT donk-specific demos. Bench harness hardcodes donk SteamID → `analyze_demo` exited early → only `construct` timed → 1.11× was meaningless. Re-ran against `for_analysis/spirit/` (Spirit team = donk's team) which DOES contain donk events. Real verdict above.

### [Rule 3 — Windows cp1252 UnicodeEncodeError on bench print]

Bench harness prints `→` (U+2192) which Python's default cp1252 stdout encoding cannot encode on Windows. Workaround: set `PYTHONIOENCODING=utf-8` per invocation. Permanent fix would be `sys.stdout.reconfigure(encoding='utf-8')` in the harness itself.
