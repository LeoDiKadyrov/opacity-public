## Summary

Phase 9 builds one new file — `report_generator.py` — and adds one `st.download_button` block at the bottom of the Interpretation section in `app.py`. No new external dependencies are required: Jinja2 3.1.6 is already available (installed as a Streamlit transitive dep), matplotlib Agg backend is already configured, and `sqlite3` cursor pattern for SteamID64 precision is already established in `interpretation.py`.

The main technical decisions that the planner needs to know: (1) `st.download_button(data=bytes)` is verified correct — Streamlit 1.54.0 accepts `bytes` directly; (2) report HTML is generated on every Streamlit rerun unless `@st.cache_data` is applied — chart generation costs ~0.12s per figure, so ~10 figures for full report = ~1.2s without caching; (3) `visualize_results.py` chart functions use non-Djok colors and read from CSV, not SQLite — `report_generator.py` needs its own chart helpers that read from `analytics.db` directly and apply brand colors.

**Primary recommendation:** Use Python f-strings for HTML generation (not Jinja2 — scope is small enough that f-string is simpler and has zero runtime overhead), write dedicated `_render_chart_b64()` helper inside `report_generator.py` with brand colors, and wrap `generate_html_report()` with `@st.cache_data(ttl=300)` to avoid re-rendering charts on every Streamlit interaction.

### Claude's Discretion: f-string vs Jinja2
**Recommendation: f-string.** The report HTML is ~300–400 lines, generated once per function call, with no conditional blocks complex enough to benefit from a template engine. Jinja2 adds a `Environment`/`Template` object and requires escaping `{` in CSS; f-string requires escaping `{` in CSS too (as `{{`/`}}`), but has zero import overhead and is simpler to read inline. If the HTML grows beyond ~600 lines, revisit.

**Installation:** No new packages needed. All dependencies already present.

### Pattern 1: matplotlib base64 encoding
**What:** Render a matplotlib figure to PNG bytes, base64-encode for HTML `<img>` src.
**When to use:** Any chart in report_generator.py.
**Example:**
```python
# Source: VERIFIED in session — works with matplotlib 3.10.8 + Agg
import io, base64
import matplotlib
matplotlib.use("Agg")
import matplotlib.pyplot as plt

def _render_chart_b64(data, metric_label: str, engagement_type: str) -> str:
    fig, ax = plt.subplots(figsize=(10, 4), facecolor="#0e0e12")
    ax.set_facecolor("#16161d")
    ax.hist(data.dropna(), bins=20, color="#e8b84b", alpha=0.85)
    ax.axvline(float(data.median()), color="#e0e0e8", linestyle="--", linewidth=1.2)
    ax.tick_params(colors="#e0e0e8")
    ax.spines["bottom"].set_color("#2a2a35")
    ax.spines["left"].set_color("#2a2a35")
    fig.tight_layout()
    buf = io.BytesIO()
    fig.savefig(buf, format="png", dpi=96, bbox_inches="tight", facecolor=fig.get_facecolor())
    buf.seek(0)
    b64 = base64.b64encode(buf.read()).decode("utf-8")
    plt.close(fig)
    return b64
```

### Pattern 4: @st.cache_data for report generation
**What:** Cache the output of `generate_html_report()` to avoid re-rendering on every Streamlit rerun.
**When to use:** Wrap `generate_html_report()` in report_generator.py.
**Example:**
```python
# Source: VERIFIED — st.cache_data available in Streamlit 1.54.0
import streamlit as st

@st.cache_data(ttl=300)  # 5-minute cache — report changes only if DB changes
def generate_html_report(player_steamid: int, benchmark_steamid: int, benchmark_name: str, db_path: str) -> bytes:
    ...
```
**Caveat:** `@st.cache_data` must be imported at module level. `report_generator.py` will import `streamlit` only for this decorator. If that creates a circular concern, use a `functools.lru_cache` alternative (but `st.cache_data` is cleaner for Streamlit contexts).

### Pattern 5: CSS with escaped braces in f-strings
**What:** Embed CSS in an f-string without Python interpreting `{}` as format expressions.
**When to use:** HTML template in `_assemble_html()`.
**Example:**
```python
# In an f-string, CSS literal braces must be doubled:
css = f"""
<style>
  body {{ background: #0e0e12; color: #e0e0e8; }}
  .card {{ border: 1px solid {accent_color}; }}
</style>
"""
# Note: {{ and }} are literal { and } in the output
```

### Anti-Patterns to Avoid
- **pd.read_sql for SteamID64 columns:** Float64 truncates 17-digit SteamID64 silently. Always use `cursor.fetchall()` + manual column extraction. Established in Phase 8.
- **Calling visualize_results.py from report_generator.py:** Those functions use different color palettes (green/red/blue) and read from CSV, not SQLite. They will produce wrong-colored charts and require CSV to be present. Write dedicated chart helpers.
- **Generating charts outside @st.cache_data:** Without caching, ~10 matplotlib figures at 0.12s each = 1.2s added to every Streamlit interaction. Cache on (player_steamid, benchmark_steamid, db_path).
- **External font/CDN links in HTML:** The report must pass the registry safety check from UI-SPEC. Font stacks use system fallbacks only — no `<link rel="stylesheet" href="https://fonts.googleapis.com/...">`.
- **Generating report only on button click:** `st.download_button` does not have an `on_click` callback that runs before serving. Data must already be in `data=`. Generate HTML before the button call, every rerun (mitigated by caching).

## Common Pitfalls

### Pitfall 1: `plt.close(fig)` not called after base64 encoding
**What goes wrong:** Matplotlib accumulates open figures in memory. 10+ figures per report × multiple reruns = memory leak in a long-running Streamlit session.
**Why it happens:** `fig.savefig()` does not close the figure.
**How to avoid:** Always call `plt.close(fig)` immediately after encoding to bytes.
**Warning signs:** Growing memory usage in `streamlit run` process.

### Pitfall 2: f-string CSS braces not escaped
**What goes wrong:** `SyntaxError` or garbled CSS output when `{` in CSS is unescaped in f-string.
**Why it happens:** Python interprets `{color}` as a format variable.
**How to avoid:** Double all literal braces: `body {{ margin: 0; }}`.
**Warning signs:** `KeyError` during report generation, or CSS rendered as Python variable names.

### Pitfall 3: `st.cache_data` not applied — slow reruns
**What goes wrong:** Every Streamlit interaction (any widget change anywhere in the app) triggers chart regeneration = 1.2s lag on every rerun.
**Why it happens:** All Python code in `app.py` reruns on every Streamlit event.
**How to avoid:** `@st.cache_data(ttl=300)` on `generate_html_report()`.
**Warning signs:** 1–2 second freeze on any Streamlit interaction after Interpretation section loads.

### Pitfall 4: `_bm_selection.split(" (")[0]` used in cache key
**What goes wrong:** Cache hit misses if benchmark display name contains " (" in unexpected places.
**Why it happens:** `benchmark_name` is derived by splitting `_bm_selection` in `app.py`.
**How to avoid:** Pass `benchmark_steamid` (int) as cache key, not display name string.
**Warning signs:** Stale report shown after benchmark player changes.

### Pitfall 5: Worst metric card shown even when tier is `n/a`
**What goes wrong:** Empty/confusing card for engagement types with no valid data.
**Why it happens:** `get_worst_metric()` can return a row with tier `n/a` if not filtered.
**How to avoid:** Check `if worst and worst.get("tier") not in ("n/a", None)` before rendering card. This guard already exists in `app.py` — replicate in `report_generator.py`.
**Warning signs:** Card appears with `n/a` tier and generic drill text.

### Pitfall 6: `dpi` not set → oversized base64 strings
**What goes wrong:** Default matplotlib dpi=100 produces larger PNG than needed; at 200dpi, file size increases 4x.
**Why it happens:** `fig.savefig()` defaults to figure.dpi (often 100 or 200 depending on rcParams).
**How to avoid:** Explicitly pass `dpi=96` to `fig.savefig()`. UI-SPEC specifies `96 dpi`.
**Warning signs:** HTML file >5MB, slow browser rendering.

### Full generate_html_report() signature
```python
# Pattern established from 09-CONTEXT.md integration spec
@st.cache_data(ttl=300)
def generate_html_report(
    player_steamid: int,
    benchmark_steamid: int,
    benchmark_name: str,
    db_path: str,
) -> bytes:
    """Return fully self-contained HTML report as UTF-8 bytes."""
    peek_rows = compute_interpretation(db_path, player_steamid, benchmark_steamid, "peek")
    hold_rows = compute_interpretation(db_path, player_steamid, benchmark_steamid, "hold")
    raw_rows, col_names = _get_raw_engagements(db_path, player_steamid)
    charts_b64 = _generate_all_charts(db_path, player_steamid)
    html = _assemble_html(
        player_steamid, benchmark_name, peek_rows, hold_rows, charts_b64, raw_rows, col_names
    )
    return html.encode("utf-8")
```
