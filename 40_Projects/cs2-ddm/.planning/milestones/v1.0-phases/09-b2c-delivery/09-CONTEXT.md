# Phase 9: B2C Delivery — Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Build a self-contained HTML report generator integrated into the existing Streamlit app. A paying user sends their .dem file manually (email/Telegram); the operator runs the pipeline locally; the Interpretation tab gains a "Download HTML Report" button that produces a single portable .html file combining interpretation tier table, inline charts (base64), and raw engagement data. No public deploy in this phase. FACEIT API approval runs in parallel but does not block delivery.

</domain>

<decisions>
## Implementation Decisions

### Deploy Strategy
- **D-01:** No server deploy in Phase 9. App stays local. Deploy = Phase 10.
- **D-02:** FACEIT API approval process starts now (30-day turnaround) — operator submits application. Does not block Phase 9 work.

### Demo Input Flow
- **D-03:** Demo delivery is manual — user sends .dem file via email or Telegram. No automated upload mechanism in this phase.
- **D-04:** SteamID provided by user at intake (Tally form already collects this). Operator enters it in Streamlit sidebar before running analysis.

### Report Format
- **D-05:** Output = single self-contained `.html` file — all CSS, JS, charts, and data inline. User opens in browser, no dependencies.
- **D-06:** Design = Djok terminal brand: `#0e0e12` background, `#e8b84b` gold accent, JetBrains Mono for numbers, borders not shadows. Consistent with landing page.
- **D-07:** Report has three sections in order:
  1. Interpretation — worst metric card (punchline first), then full 5-metric tier table vs donk
  2. Charts — distribution plots per metric (matplotlib figures encoded as base64 PNG inline)
  3. Raw data — full engagements table (all moments, all columns)

### Report Generation Entry Point
- **D-08:** `st.download_button` added to Interpretation tab in `app.py`. Generates HTML in-memory, serves as download. No new CLI script needed.
- **D-09:** HTML generated by new `report_generator.py` module. Called from `app.py` — keeps report logic separate from Streamlit UI.

### Email Gate
- **D-10:** Tally form on landing page is unchanged — already collects SteamID + email. No new email infrastructure.
- **D-11:** Operator workflow: Tally submission → receive demo → run pipeline → download HTML → send to user email manually.

### Claude's Discretion
- HTML template structure (Jinja2 vs f-string vs inline Python string concat)
- Chart sizing and layout within HTML sections
- Raw data table column ordering and formatting (highlight key columns vs show all)
- Exact wording of section headers in report

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `interpretation.py:compute_interpretation()` — returns tier data per metric; HTML report calls same function.
- `interpretation.py:get_worst_metric()` — already computes punchline; use for "Your worst metric is X" header card.
- `visualize_results.py` — matplotlib figure generation already works in `Agg` mode (non-interactive); encode with `io.BytesIO` + `base64.b64encode()` for inline HTML.
- `app.py` sidebar `steamid_input` — SteamID already in session state; report generator reads it directly.

### Established Patterns
- `matplotlib.use("Agg")` already set in `app.py` — safe to generate figures server-side and encode to base64.
- `sqlite3` cursor pattern from Phase 8 fix (cursor.fetchall() for SteamID64 precision) — reuse in report_generator.py, never `pd.read_sql` for SteamID columns.
- `engagement_type` split (peek/hold) already used in Interpretation tab — report must preserve same split.

### Integration Points
- `app.py` Interpretation tab → import `report_generator.generate_html_report(steamid, benchmark_steamid, db_path)` → `st.download_button(data=html_bytes, file_name="djok_report_{steamid}.html")`
- `report_generator.py` → calls `interpretation.py` + `visualize_results.py` + direct sqlite3 read → returns bytes

</code_context>

<specifics>
## Specific Ideas

- Report filename: `djok_report_{steamid}.html` — includes SteamID for traceability when emailing.
- Worst metric card is the FIRST thing in the report (punchline before data) — matches SC5 and Phase 8 D-02 decision.
- Charts should be the same distributions shown in the Streamlit app — user gets a static snapshot of their session.

</specifics>

<deferred>
## Deferred Ideas

- **Public deploy (Railway/Render/VPS)** — Phase 10. No budget now; architecture decisions pending.
- **FACEIT URL auto-download** — blocked on API approval (30-day). Phase 10+.
- **Unique shareable URL** — requires server-side state. Phase 10 after deploy.
- **Automated email delivery** (Resend/Mailchimp) — Phase 10; Tally + manual send is sufficient for early access volume.
- **PDF export** — weasyprint/pdfkit adds complexity. HTML is sufficient for MVP; revisit if users request.
- **st.download_button in public deploy** — will work as-is when deploy happens; no changes needed for Phase 10 to pick it up.

</deferred>
